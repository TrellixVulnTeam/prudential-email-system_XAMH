Typography

Headings

Header tags are styled with a class that matches the heading level, so they can be adjusted on the fly.

If you want style headings differently, use the heading classes .h1, .h2, .h3, .h4

<h2 class="h2">This is an H2 header. Set up a section of the email.</h2>

Accessibility tip: Headers must maintain their hierarchy order.

H1 Tag

If using an H1 tag with a subhead, we style the subhead with a <span> within the <h1> tag, to keep it compliant for accessibility.

Be careful to not break the H1 header tag! There can only be one <h1> tag per email, so make sure the subhead is inside the <h1>.

Accessibility tip: Every email must have an <h1> tag. There can only be one <h1> tag per email.

Bullets

Bullets can be done just like standard html bullets.

Other typography classes

.disclosure is used for any regular-sized disclosure.

.disclosure-large is for disclosure that needs to appear larger than the rest of the disclosure.

.superscript is placed on a <span> tag around any text that must appear superscripted.

Most modules will come with the class="section" already applied. You can replace it with one of the following:

.section-bottom-large This will adjust the bottom spacing of a module to be larger

Mobile helper classes

Some of these are used strategically around various modules, and may come in handy if you want to do the following things:

.center-content-on-break overrides the text spacing, useful for multi-column elements that are right aligned on desktop but look crazy on mobile.

.emailwrapto100pc will cause smaller images to become full width on mobile. In some components, this class is not used by default. Without it, images will retain their original dimensions.

.emailwrapto100pcbreak is used on a <div> or an <img> to make them stretch to full width on mobile devices and ensure that other items break underneath them

.emailwrapto50pcbreak is used on an <img> to make them stretch to 50% width on mobile devices

.mobile-hide will cause elements to dissapear only on mobile devices

.mobile-only will cause elements to show only on mobile devices

Images

When placing images in email in building utilities like Litmus or Dreamweaver, they usually get placed at full size. A few tips to get the image working correctly:

  • Make note of the image width in the component you are using, and ensure that your new image dimensions are no larger than this.
  • Make sure the image has height="auto", so it will scale properly. No need to do math and figure out how to scale both the width and height at the same time.
  • Design teams will usually create images for email "@2x", which means they contain twice as many pixels as needed in order to look crisp on high-resolution displays such as iPhones. Simply divide the width in half, and the image should look great.

Always test for Dark Mode clients! Dark mode in Outlook/O365/Office.com and other clients are quite destructive, and can cause some images to appear hard to see against their backgrounds. Although this cannot be fixed in code, the images may have to go back to the design team for optimization.

Accessibility tips:

  • Alt text should only be used if the image has some sort of text content that a visually impaired user needs to read. Although, text in images is discouraged when it can be avoided.
  • <img> tags should always have an alt tag, even if it is empty. An empty alt="" tells a screen reader that an image is decorative, and can be ignored.

Coding tips

Proper table structure

Pay attention to your table structure when pasting in or removing content! If an email is testing improperly, it usually means something is in the wrong place, or table structure is incomplete. Table structure will always go <table> > <tr> > <td>.

Nested tables are always inside a <td>.

Resetting table borders and padding

Every <table> should always have border="0" cellspacing="0" cellpadding="0" on them, in order to prevent any unwanted borders, spacing, or layout issues.

Code comments

Code comments are your friend. They will help you determine where sections start and end, or where to add and remove code.

MSO code

Watch out for MSO code! Throughout various parts of the email system, whenever you see code that starts with [if (gte mso 9)|(IE)], then that code is meant to affect changes only on Outlook email clients. In the rare occasion when adjusting things like email background colors or column widths, you may have to adjust the codes in here as well or it may not appear correctly in Outlook.

How to manually break text

Every now and then, we will want to break a line manually in order to maintain prettier typography design, or to keep product names and phone numbers on one line. However, when this is done with hard line breaks (inserting a <br> or hitting the return key in Dreamweaver), it can cause unwanted text breaks on smaller devices. The example below shows how to insert a <br> tag to break the line, inside of a <span> that will dissapear on mobile devices.

No break:

To learn more about FlexGuard, call the Sales Desk at XXX-XXX-XXXX.

Mobile with forced line break:

To learn more about FlexGuard, call the Sales Desk
at XXX-XXX-XXXX.

Desktop with break:

To learn more about FlexGuard, call the Sales Desk
at XXX-XXX-XXXX.

Mobile with hidden break:

To learn more about FlexGuard, call the Sales Desk at XXX-XXX-XXXX.
